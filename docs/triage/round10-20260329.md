## Coder is doing now

### Fix 1: Wrap individual provider calls to catch rejections

**Source:** Codex HIGH
**Safe to apply now** — defensive, no behavior change for well-behaved providers

`getRatesFromAll()` uses `Promise.all(providers.map(p => p.getRates(request)))`.
If any provider's `getRates()` rejects (throws instead of returning Result),
`Promise.all` rejects and `getRatesFromAll` itself throws — violating the
boundary contract ("never throws").

Third-party carrier implementations might throw. The registry is the extension
point; it must be defensive at the boundary.

**Fix:** Wrap each call:
`p.getRates(request).catch(err => ({ ok: false, error: String(err) }))`

### Fix 2: Export CarrierRegistry from index.ts

**Source:** Claude LOW + Codex MEDIUM (2-family consensus)
**Safe to apply now** — no behavior change

The registry is the public extension point for step 11 but isn't accessible
through the package boundary.

### Fix 3: Connect UpsRateProvider to CarrierProvider

**Source:** Claude MEDIUM
**Safe to apply now** — compile-time only, no runtime change

This is the structural verification that step 11 exists to establish.
Add a `satisfies` assertion in carrier-ups to verify UpsRateProvider
implements CarrierProvider.

## Design decision needed

### Partial failure semantics in getRatesFromAll

**Source:** Claude MEDIUM + Gemini HIGH (2-family consensus)

When UPS succeeds but FedEx fails, the caller gets `{ ok: true, data: [UPS quotes] }`
with no indication that FedEx failed. The error is collected but discarded.

Options:
- **(a)** Accept current behavior — MVP, caller gets best-available quotes
- **(b)** Extend return type: `{ ok: true, data: quotes, warnings: errors[] }`
- **(c)** Return `{ ok: false }` with partial data when not all carriers respond

This is a business decision, not a code fix. Worth a devlog entry.
