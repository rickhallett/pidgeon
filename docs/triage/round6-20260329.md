## Coder is doing now

### Fix 1: Retry on timeout (AbortError) instead of returning immediately

**Source:** Claude CRITICAL (single-family, but high confidence)
**Blocked by:** Tester T1 (needs test update first)

`rate.ts:66-67` — when fetch times out, the AbortError handler does a hard
`return`, exiting the retry loop. Every other transient failure (network error,
500, 502, 503, 429) sets `lastResult` and `continue`s. Timeouts are the most
common transient failure in HTTP integrations (latency spikes, TLS handshake
delays, server GC pauses). The code inverts retry priority: rare persistent
failures get retried, common transient failures don't.

**Fix:** Change `return` to `lastResult = ...; continue;`:
```typescript
if (error instanceof DOMException && error.name === "AbortError") {
  lastResult = { ok: false, error: "Request timeout" };
  continue;
}
```

### Fix 2: Simplify timeout to native AbortSignal (eliminate Promise.race)

**Source:** Claude HIGH + Gemini LOW (cross-family consensus on the concern)
**Blocked by:** Tester T2 (fake fetch must respect signal for test validity)

The Promise.race pattern races fetch against a manual abort-listener promise.
This creates orphaned promises on the success path (the abort-listener promise
is never settled), leaks event listeners, and risks unhandled promise rejections
in edge cases. The pattern exists to work around fake fetches that don't respect
AbortSignal — but the real fix is to make the fakes respect the signal.

Bun's fetch natively rejects with AbortError when the signal fires. Simplify to:
```typescript
const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), timeoutMs);
try {
  response = await this.config.fetch(url, { ...options, signal: controller.signal });
} finally {
  clearTimeout(timeoutId);
}
```

**Dependency:** Fix 1 must land first (timeout must be retriable before we
change how timeout is signalled).

## Design decision needed

### 401 from rating endpoint: reacquire token + retry once?

**Source:** Codex HIGH
**Context:** PRD says token refresh should be transparent to callers.

Current behavior: 401 clears cached token, returns error. Caller must retry.
Proposed behavior: 401 clears token, reacquires once, retries the rating call
once. If second attempt also fails, return error.

**Trade-offs:**
- Pro: Transparent recovery matches PRD requirement
- Pro: Token expiry between acquisition and use is a realistic race condition
- Con: Adds complexity to retry loop (auth retry vs HTTP retry are different concerns)
- Con: Current retry tests assert 401 is never retried — must update

**Recommendation:** Defer to a dedicated step after HTTP hardening is complete.
This is a design decision, not a bug fix. The 401 invalidation (round 4 fix)
already handles the cache side. The retry-once behavior can be added as a
focused change with its own test.
