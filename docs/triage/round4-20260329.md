## Coder is doing now

### Fix 1: Invalidate cached token on 401 from rating endpoint

**Source:** Claude CRITICAL + Codex HIGH (cross-family consensus)
**Blocked by:** Tester T1 (needs failing test first)

When the rating endpoint returns 401 ("Invalid Access Token"), `handleHttpError`
returns a Result error but `cachedToken` is never cleared. If a token is revoked
server-side before `expires_in` elapses, every subsequent `getRates()` call reuses
the revoked token — failing with the same 401 for up to 4 hours.

**Fix:** Set `this.cachedToken = null` when rating endpoint returns 401, before
returning the error. Tester provides failing test (T1), then coder implements.

### Fix 2: Add 60-second expiry buffer for clock skew

**Source:** Gemini MEDIUM + Claude MEDIUM (cross-family consensus)
**Safe to apply now** — defensive, no observable behavior change

`Date.now() < expiresAt` compares local time against a locally-computed deadline,
but network latency means the token was issued seconds before the local timestamp.
Standard practice is a 30-60 second buffer.

**Fix:** `expiresAt: Date.now() + Math.max(0, expiresIn - 60) * 1000`

### Fix 3: Parse expires_in flexibly (number or string)

**Source:** Claude MEDIUM
**Safe to apply now** — defensive, no observable behavior change

`typeof body.expires_in === "number"` rejects string values. UPS uses
string-encoded numerics throughout the Rating API. If the OAuth endpoint
also returns `"14399"` instead of `14399`, the fallback to `0` causes every
`getRates()` call to re-acquire a token — silent performance degradation.

**Fix:** `const expiresIn = typeof raw === "number" ? raw : parseInt(String(raw), 10) || 0`
