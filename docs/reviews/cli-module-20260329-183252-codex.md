[HIGH] Validation failures are silent in the actual CLI

Where:
- `packages/core/src/cli.ts:12`
- `packages/core/src/cli.ts:17`
- `packages/core/src/cli.test.ts:235`

What:
`createProgram()` disables all Commander output with `configureOutput({ writeErr: () => {}, writeOut: () => {} })`. When required flags are missing, Commander still throws via `exitOverride()`, but the user gets no validation message on stderr/stdout.

Why it matters:
The CLI is supposed to be usable dev tooling (`spec.md:82-85`, `BUILD_ORDER.md:50-52`). In the current implementation, a malformed invocation fails silently unless the caller is catching Commander exceptions programmatically. The existing test claims to verify that a validation error is displayed, but it only checks that parsing throws.

Evidence:
A direct run of `createProgram()` with `--origin-postal` omitted throws Commander’s `missingMandatoryOptionValue` error while producing no captured output.

Recommendation:
Either let Commander write its validation errors normally, or route `writeErr` through an injected error writer and assert on the rendered message in tests.

[HIGH] Provider failures do not produce a non-zero CLI failure path, and the test does not actually check that

Where:
- `packages/core/src/cli.ts:70`
- `packages/core/src/cli.test.ts:209`

What:
On provider failure the action only writes `Error: ...` and returns. It never sets `process.exitCode`, calls `program.error()`, or otherwise signals command failure. The corresponding test is named “sets non-zero exit code on provider failure” but only asserts that the output contains the error text.

Why it matters:
For a CLI, human-readable output is not enough. Shell scripts and CI callers depend on a failing exit status. As written, `pidgeon rate ...` can emit an error and still exit successfully, which makes the command unreliable for automation.

Evidence:
A direct invocation with a provider returning `{ ok: false, error: "Network timeout" }` completes without throwing and with only `["Error: Network timeout"]` captured.

Recommendation:
Decide on a real failure contract for provider errors, implement it, and update the test to assert the actual exit behavior instead of just the message.

[MEDIUM] Numeric flags are accepted without validation and can reach the provider as `NaN`

Where:
- `packages/core/src/cli.ts:48`
- `packages/core/src/cli.ts:50`
- `packages/core/src/cli.test.ts:84`
- `spec.md:41-49`

What:
The CLI converts numeric options with `Number(...)` and forwards the result directly into `RateRequest`. Invalid values like `--weight abc` become `NaN` and are still sent to `provider.getRates()`. The current tests only cover well-formed numeric strings, so this boundary is untested.

Why it matters:
The spec says runtime validation should happen at the service boundary before external calls. Right now the CLI can construct an invalid request shape and rely on downstream code to notice it, which weakens the boundary and makes failures less predictable.

Evidence:
A direct invocation with `--weight abc` reaches the fake provider with `request.packages[0].weight.value === NaN` (serialized as `null` in the probe output).

Recommendation:
Parse and validate numeric flags in the CLI before calling the provider, and add tests that reject non-numeric or non-positive dimensions/weights.
