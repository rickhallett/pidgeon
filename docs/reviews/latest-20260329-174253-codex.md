## Findings

### High: Expired or invalid cached bearer tokens are still terminal instead of transparently refreshed

[`packages/carrier-ups/src/rate.ts`](/Users/mrkai/code/pidgeon/packages/carrier-ups/src/rate.ts#L73) treats a `401` from the rating endpoint as a final error and returns immediately after clearing the cache in [`packages/carrier-ups/src/rate.ts`](/Users/mrkai/code/pidgeon/packages/carrier-ups/src/rate.ts#L125). There is still no one-time reacquire-and-retry path when a cached token expires between token lookup and the outbound rating call.

Why it matters: the repo requirements say token refresh on expiry should be transparent to callers in [`docs/PRD.md`](/Users/mrkai/code/pidgeon/docs/PRD.md#L49). The new retry suite actually cements the opposite behavior by asserting that `401` is never retried in [`packages/carrier-ups/src/http-retry.test.ts`](/Users/mrkai/code/pidgeon/packages/carrier-ups/src/http-retry.test.ts#L303).

Recommendation: distinguish auth-expiry recovery from general client-error retry policy. On a rating `401`, invalidate the cached token, reacquire once, and retry the request once.

### Medium: `Retry-After` parsing only handles integer seconds, not valid HTTP-date values

[`packages/carrier-ups/src/rate.ts`](/Users/mrkai/code/pidgeon/packages/carrier-ups/src/rate.ts#L77) parses `Retry-After` with `parseInt`. That works for delta-seconds like `"1"`, but `Retry-After` also allows HTTP-date values. In that case `parseInt` yields `NaN`, the header is effectively ignored, and the client falls back to exponential backoff instead of respecting the serverâ€™s rate-limit instruction.

Why it matters: the HTTP-layer requirement explicitly calls for `Retry-After` parsing in [`spec.md`](/Users/mrkai/code/pidgeon/spec.md#L68). The tests only cover the numeric form in [`packages/carrier-ups/src/http-retry.test.ts`](/Users/mrkai/code/pidgeon/packages/carrier-ups/src/http-retry.test.ts#L270), so this realistic variant would ship untested.

Recommendation: support both delta-seconds and HTTP-date forms, and add one test for each.

### Medium: Retry policy is hardcoded in library code instead of being configurable

[`packages/carrier-ups/src/rate.ts`](/Users/mrkai/code/pidgeon/packages/carrier-ups/src/rate.ts#L28) hardcodes `maxAttempts`, `baseDelayMs`, `timeoutMs`, and the `Retry-After` cap directly inside `getRates()`.

Why it matters: the spec calls for exponential retry with configurable max attempts in [`spec.md`](/Users/mrkai/code/pidgeon/spec.md#L66). Hardcoding the policy means tuning for production behavior requires code edits and retests rather than configuration changes.

Recommendation: move retry/timeout policy into provider configuration or a small internal transport config object.
